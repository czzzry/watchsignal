# Decision Register

This register captures the major decisions that shape the project.

Research recommendations below are provisional inputs from `docs/research/n8n deep research.md`, not accepted truth.
The founder has now made initial working decisions for the first vertical slice.
These are accepted as founder direction for planning and reversible scaffolding.
They are not yet accepted ADRs.

| Decision area | Options | Tradeoffs | Research recommendation | Reversibility | Revisit triggers | Founder decision |
|---|---|---|---|---|---|---|
| Product framing | Full movie app, thin household mediator, household decision assistant | Broader product scope can feel impressive but duplicates existing products and increases maintenance; a thinner mediator stays focused on the unique couple decision problem | Thin household mediator layered over public movie metadata and simple household rules | High | Desire for public profiles, reviews, discovery feeds, or a standalone content app | Private household mediator focused on improving shared recommendations over time, not a public movie app |
| Build vs buy | Use existing consumer apps only, build custom mediator, hybrid | Buying is faster and lower effort; building creates portfolio value and supports custom consensus logic; hybrid reduces reinvention | Build the mediator but buy or reuse the metadata layer instead of rebuilding a movie platform | Medium | Discovery of an existing tool that already handles two-person consensus, fairness, and exportable household data | Build the custom mediator and buy the metadata layer |
| Interface platform | Telegram, WhatsApp, Discord, Slack, n8n Forms, lightweight web UI | Chat is convenient but platform-specific; forms are simple but less conversational; richer UI increases build scope | Telegram first because it is bot-friendly and well-aligned with n8n orchestration | High | Refusal to use Telegram, need for richer UI controls, or platform friction in real usage | Telegram first for MVP, with separate-device mode primary and pass-the-phone mode also supported |
| Movie data source | TMDb, OMDb, Trakt, Letterboxd import/export, availability providers, scraping | Rich official APIs are cleaner; thinner sources limit metadata; scraping adds fragility and risk; some platforms have access restrictions | TMDb as the primary metadata source, with availability enrichment deferred | Medium | Need for stronger streaming availability coverage, personal watch-history sync, or regional availability accuracy | TMDb first, while explicitly validating Amazon.de availability and English audio or foreign plus English subtitles |
| Storage layer | n8n internal state, Google Sheets, Airtable or Baserow, SQLite, Supabase or Postgres, Firebase | Visible tools are easier to debug; app databases scale better later; some options are awkward with n8n or add unnecessary ops early | Google Sheets first, with Supabase or Postgres as the first serious upgrade path | High | Painful joins, concurrent writes, audit needs, or desire for SQL analytics and stronger data integrity | Google Sheets first for n8n learning, visibility, and fast iteration, with a later migration path to Supabase or Postgres |
| Recommendation method | Manual shortlist, popularity plus filters, metadata scoring, content similarity, collaborative filtering, hybrid, bandit | Simpler methods are easier to explain and debug; advanced methods need more data and experimentation discipline | Metadata scoring with transparent per-user scoring and explicit couple aggregation | Medium | Enough feedback history to justify experiments, clear stagnation in quality, or a portfolio goal centered on recommender experimentation | Version one uses the simplest modular scoring engine that can credibly learn, with algorithmic sophistication as the long-term north star and a planned MVP plus 1 upgrade lane |
| Feedback mechanism | Binary like or dislike, Loved or Fine or No, stars, pairwise comparisons, passive completion only | Richer feedback can improve modeling but adds friction and inconsistency; simpler feedback is easier to collect reliably | Loved or Fine or No after watching, with more granularity deferred | High | Current labels prove too coarse for tie-breaks, trend analysis, or user satisfaction tracking | Onboarding leans on manual seed ratings; shortlist reactions use Interested or Maybe or No; post-watch feedback uses Loved or Fine or No; optional free-text is stored from day one |
| Couple consensus rule | Average score, additive sum, least misery, approval voting, fairness rotation, alternating pick rights | Additive scoring can steamroll one partner; stricter veto rules can become too conservative; fairness rules add ceremony | Least-misery floor with additive tie-break as the default compromise rule | High | One partner feels routinely under-served, or safe but boring picks dominate outcomes | Session mode is explicit for husband-first, wife-first, or compromise; compromise optimizes for strongest shared like with dislike protection; fairness defaults are remembered but always visible and overrideable |
| LLM role | No LLM, mood parsing only, explanation support, ranking authority, agentic recommender | LLMs help with fuzzy input and explanations but add opacity, cost, and hallucination risk if given ranking authority | Limit LLM use to mood-to-filter translation, clarifying questions, and explanations | High | Free-text interaction proves essential, or deterministic filters fail to capture real intent | MVP keeps LLM use to parsing, optional clarification, and explanations; MVP plus 1 adds LLM-assisted feedback interpretation; a mature version may use more LLM-derived taste interpretation |
| Privacy model | Minimal private IDs, richer personal history, public sanitized portfolio clone, raw household logs | Rich data can help personalization but increases privacy exposure; public artifacts need careful sanitization | Minimize stored personal text, keep private identifiers out of the repo, and use synthetic public artifacts | Medium | Expanded user scope, public demos, screenshots, or sharing outside the household | Keep the repo private first, store personal data minimally, capture optional free-text safely, and publish only sanitized artifacts if the project is later shown publicly |
| n8n vs code boundary | n8n only, n8n plus tiny code nodes, n8n plus external scorer service | n8n-only is easier to operate early; too much logic inside workflows becomes brittle; external services add maintenance but help testing | Keep orchestration in n8n and limit code to small deterministic transforms unless a real boundary is needed | Medium | Scoring logic grows large, needs libraries or tests, or workflows become hard to inspect | n8n orchestrates, but scoring lives behind a separate testable module from day one so it can be replaced without rewriting the whole flow |
| Repo/public portfolio strategy | Private repo only, sanitized architecture repo, public template/starter, full personal history | Public visibility helps portfolio value but increases sanitization work and privacy risk | Publish system design, decision rationale, and fake examples rather than personal data | High | Decision to open-source a template, share screenshots, or use the project in applications publicly | Private working repo first; later derive sanitized diagrams, docs, and examples only if the project becomes worth sharing |
| Deployment model | n8n Cloud, self-hosted n8n, hybrid, local-only | Cloud reduces ops; self-hosting gives control but adds maintenance; local-only limits convenience | Keep deployment undecided until interface, storage, and privacy choices are confirmed; current research leans toward a boring hosted default for early learning | Medium | Cost concerns, privacy constraints, need for local integrations, or desire for deeper infrastructure control | Start on n8n Cloud if convenient, but treat portability to self-hosted n8n as a first-class constraint from day one |
| External paid vendors | No paid vendors, minimal paid API only, multiple managed tools | Paid services can accelerate delivery but raise cost, lock-in, and governance burden | Defer paid vendors unless a clear product need appears; start with the minimum viable vendor set | High | Hitting hard limits on free tiers, legal or reliability needs, or a founder decision to optimize for speed over cost control | Defer paid vendors unless a concrete gap appears and the user value clearly justifies them |
